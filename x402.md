# x402 Paywall Implementation

This project uses x402 to protect selected Nuxt Content articles behind an HTTP `402 Payment Required` flow. Public articles continue to render normally. Articles marked as exclusive are served through a Nitro API endpoint that verifies and settles x402 payments before returning the full article payload.

## Scope

The current implementation protects article content only.

- Public route: `/article/:slug`
- Paid API route: `/api/articles/:slug`
- Protected content source: `content/article/*.md`
- Payment scheme: `exact`
- Default test network: Base Sepolia, `eip155:84532`
- Default facilitator: `https://x402.org/facilitator`

The browser page renders a local paywall panel with a `Pay with wallet` action. The action opens a modal containing the x402 in-browser wallet payment UI from `@x402/paywall`. Programmatic clients or agents can call the same API route with an x402-compatible payment flow and a `PAYMENT-SIGNATURE` header.

## Dependencies

Runtime dependencies are declared in `package.json`:

```json
{
  "@x402/core": "^2.13.0",
  "@x402/evm": "^2.13.0",
  "@x402/paywall": "^2.13.0",
  "react": "^19.2.6",
  "react-dom": "^19.2.6"
}
```

`@x402/core` provides the resource server, HTTP server, facilitator client, and payment processing primitives. `@x402/evm` provides the EVM `exact` scheme implementation. `@x402/paywall` provides the embedded browser wallet payment page; it depends on React.

## Configuration

Server-side runtime config is defined in `nuxt.config.ts`:

```ts
runtimeConfig: {
  x402: {
    receivingWallet: process.env.X402_RECEIVING_WALLET || '',
    facilitatorUrl: process.env.X402_FACILITATOR_URL || 'https://x402.org/facilitator',
    network: process.env.X402_NETWORK || 'eip155:84532',
    articlePrice: process.env.X402_ARTICLE_PRICE || '$0.001'
  }
}
```

Environment variables are documented in `.env.example`:

```bash
X402_RECEIVING_WALLET=
X402_FACILITATOR_URL=https://x402.org/facilitator
X402_NETWORK=eip155:84532
X402_ARTICLE_PRICE=$0.001
```

`X402_RECEIVING_WALLET` is required for exclusive articles. If it is missing, exclusive article API requests fail with `500 Missing X402_RECEIVING_WALLET`.

## Marking Content As Exclusive

Add `exclusive: true` to article frontmatter. Optionally set `price` per article.

```yaml
---
title: Example Paid Article
description: Paid content example.
date: 2026-05-26
exclusive: true
price: $0.001
---
```

If `price` is not set on the article, the API uses `runtimeConfig.x402.articlePrice`.

## Request Flow

1. The article page at `pages/article/[slug].vue` calls `/api/articles/:slug` with `useFetch`.
2. The API route loads the article with `serverQueryContent(event, /article/:slug).findOne()`.
3. If the article is public, the route returns `{ article }`.
4. If the article has `exclusive: true`, the route calls `serveExclusiveArticle`.
5. `serveExclusiveArticle` builds an x402 route config for the current API path.
6. If no valid payment is present, x402 returns `402` plus a `payment-required` header and preview metadata. Browser requests that accept HTML receive the embedded wallet payment UI.
7. If a valid `PAYMENT-SIGNATURE` is present, x402 verifies the payment, settles it, adds settlement headers, and returns `{ article }`.
8. The page renders either `ContentRenderer` for paid/public content or the paywall panel for `402`.
9. Clicking `Pay with wallet` opens the protected API endpoint in an iframe modal, which receives the HTML x402 paywall.
10. The modal locks background scrolling while it is open and can be dismissed with the backdrop or the `Close` button.
11. After payment, the iframe loads the paid JSON response. The parent page reads the response, updates the article payload, and closes the modal.

## Server Route

The Nitro route is `server/api/articles/[slug].get.ts`.

Responsibilities:

- Resolve the slug from `event.context.params`.
- Load the matching Nuxt Content article.
- Return public articles directly.
- Delegate exclusive articles to `serveExclusiveArticle(event, article, runtimeConfig.x402)`.

The route imports `serverQueryContent` from `#content/server`; client-side `queryContent` is not available in Nitro route files.

## x402 Utility

The implementation lives in `server/utils/x402-paywall.ts`.

Key parts:

- `H3X402Adapter` adapts Nitro/H3 request data to x402's `HTTPAdapter` interface.
- `serveExclusiveArticle` creates a per-request x402 resource server for the exact article endpoint.
- The HTTP resource server registers the EVM paywall provider created with `createPaywall().withNetwork(evmPaywall).build()`.
- `writeX402Response` applies x402 status codes and headers to the H3 response.
- `getArticlePrice` resolves article-specific price before falling back to global config.

The x402 route config uses:

```ts
{
  accepts: {
    scheme: 'exact',
    price: articlePrice,
    network,
    payTo
  },
  description: article.description || article.title || 'Exclusive article',
  mimeType: 'application/json'
}
```

The resource server is registered with:

```ts
new x402ResourceServer(
  new HTTPFacilitatorClient({ url: facilitatorUrl })
).register(network, new ExactEvmScheme())
```

## API Responses

Public article:

```http
GET /api/articles/hello-markdown-blog
HTTP/1.1 200 OK
Content-Type: application/json
```

```json
{
  "article": {
    "_path": "/article/hello-markdown-blog",
    "title": "Hello Markdown Blog"
  }
}
```

Exclusive article without payment:

```http
GET /api/articles/x402-integration
HTTP/1.1 402 Payment Required
Content-Type: application/json
payment-required: <base64-encoded-x402-payment-requirements>
```

```json
{
  "error": "payment_required",
  "article": {
    "title": "x402 Integration",
    "description": "A practical TypeScript integration note for charging or paying per HTTP request with x402.",
    "date": "2026-05-26",
    "exclusive": true,
    "price": "$0.001"
  },
  "x402": {
    "price": "$0.001",
    "network": "eip155:84532",
    "facilitatorUrl": "https://x402.org/facilitator"
  }
}
```

Exclusive article with a valid payment:

```http
GET /api/articles/x402-integration
PAYMENT-SIGNATURE: <base64-encoded-payment-payload>
```

```http
HTTP/1.1 200 OK
PAYMENT-RESPONSE: <base64-encoded-settlement-response>
Content-Type: application/json
```

```json
{
  "article": {
    "_path": "/article/x402-integration",
    "title": "x402 Integration",
    "exclusive": true,
    "body": {}
  }
}
```

## Page Rendering

`pages/article/[slug].vue` fetches the API route instead of reading content directly.

- `articlePayload.value?.article` renders through `ContentRenderer`.
- A `402` fetch error is converted into a `paywall` computed value.
- Non-`402` failures still throw Nuxt errors.
- SEO metadata falls back to the preview metadata returned in the `402` body.

The paywall panel displays:

- Article title
- Description
- Price
- Network
- API endpoint
- `Pay with wallet` action for modal browser wallet checkout
- `Copy endpoint` action for programmatic clients
- Required protocol/header hint

## Modal Checkout UI

The modal checkout is implemented in `pages/article/[slug].vue` using Vue state and `Teleport`.

State and handlers:

- `isPaywallModalOpen` controls modal visibility.
- `paywallFrameKey` forces a fresh iframe request each time the modal opens.
- `paywallFrame` references the iframe so the parent page can inspect the loaded document.
- `openPaywallModal` opens the modal and refreshes the iframe.
- `closePaywallModal` closes the modal.
- `handlePaywallFrameLoad` watches iframe loads. If the loaded document body is JSON with an `article` property, it treats that as a successful paid response, assigns it to `articlePayload`, clears `articleError`, and closes the modal.

The iframe points at the same protected API endpoint shown in the panel:

```vue
<iframe
  :key="paywallFrameKey"
  ref="paywallFrame"
  class="paywall-modal-frame"
  :src="paywallEndpoint"
  title="x402 wallet payment"
  @load="handlePaywallFrameLoad"
/>
```

The modal is same-origin because it loads `/api/articles/:slug`, so the parent page can read the final JSON response after a successful payment. The handler catches unreadable intermediate states because wallet or blob transitions may temporarily produce documents that are not accessible or are not JSON.

Styling is defined in `assets/css/tailwind.css`:

- `.paywall-actions` uses `mb-8 mt-6` to keep clear spacing between the payment buttons and the explanatory text below.
- `.paywall-action-primary` uses the accent background with white text for `Pay with wallet`.
- `.paywall-modal` creates a fixed full-screen overlay.
- `.paywall-modal-panel` constrains the checkout to `max-w-3xl` and `92svh`.
- `.paywall-modal-frame` fills the remaining panel height and renders the x402 paywall page.

## Article Index

`pages/article/index.vue` labels exclusive articles in the listing:

```vue
{{ article.exclusive ? 'Exclusive / ' : '' }}{{ article.date ?? 'Undated' }}
```

The index still links to `/article/:slug`; the article page handles whether the content is public or protected.

## Current Limitations

- No user account or entitlement persistence exists.
- Each paid request unlocks only that API response.
- Payment server setup is created per exclusive request; this is simple but can be optimized later with cached server instances.
- Only EVM `exact` payments are registered.
- Only article content is protected.
- The content body is returned after payment as JSON through the API; direct markdown files are still part of the repository/build artifact.

## Verification

Build:

```bash
npm run build
```

Manual API checks:

```bash
curl -i http://127.0.0.1:3000/api/articles/hello-markdown-blog
curl -i http://127.0.0.1:3000/api/articles/x402-integration
curl -i -H 'Accept: text/html' -H 'User-Agent: Mozilla/5.0' http://127.0.0.1:3000/api/articles/x402-integration
```

Expected behavior:

- Public article returns `200`.
- Exclusive article without payment returns `402` and a `payment-required` header.
- Exclusive article with browser HTML headers returns a `402` HTML paywall page containing `window.x402`.
- Exclusive article page renders a paywall panel instead of throwing a Nuxt error.
- `Pay with wallet` opens the x402 checkout modal without navigating away from the article page.

## Production Notes

Before production:

1. Replace `X402_RECEIVING_WALLET` with a real receiving wallet.
2. Decide whether to stay on testnet or change `X402_NETWORK` to a production-supported network.
3. Confirm the facilitator supports the selected network and scheme.
4. Consider entitlement persistence if one payment should unlock repeat reads.
